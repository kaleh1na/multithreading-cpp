# PSP — Payment Service Provider

## 1. Описание проекта

Payment Service Provider — клиент-серверная система обработки платежей между участниками: плательщиками и мерчантами.

Система обеспечивает:
- регистрацию аккаунтов двух типов: плательщик (`payer`) и мерчант (`merchant`),
- пополнение баланса аккаунта,
- выполнение платежа — атомарное списание средств с аккаунта плательщика и зачисление на аккаунт мерчанта,
- просмотр текущего баланса,
- просмотр истории транзакций по аккаунту.

Ключевое требование — корректная обработка конкурентных платежей: при одновременном списании с одного аккаунта несколькими клиентами баланс не должен уйти в минус, каждая транзакция получает корректный статус.

---

## 2. Архитектура

Система состоит из двух независимых компонентов: сервера и клиента.

### 2.1 Сервер

Реализован на C++ с использованием Boost.Asio. Принимает TCP-соединения и обрабатывает запросы пулом потоков.

**Многопоточность сервера:**

Входящие платёжные запросы помещаются в очередь задач (`TaskQueue`). Сетевой обработчик создаёт транзакцию со статусом `pending`, сохраняет её и возвращает `tx_id` клиенту немедленно. Воркеры извлекают задачи из очереди и выполняют их параллельно. Очередь защищена `std::mutex`, воркеры ожидают новых задач через `std::condition_variable` — классический event loop.

Клиент может запросить статус транзакции по `tx_id` отдельным запросом `GET_TX_STATUS`.

Синхронизация при выполнении платежа:
- На каждый аккаунт заведён отдельный `std::mutex`.
- При переводе между двумя аккаунтами оба мьютекса захватываются одновременно через `std::scoped_lock` — это исключает deadlock при встречных переводах (A→B и B→A из разных потоков).
- Хранилище аккаунтов (`AccountStore`) защищено `std::shared_mutex`: поиск аккаунта выполняется через `std::shared_lock` (параллельно из нескольких воркеров), добавление нового аккаунта — через `std::unique_lock`.
- Глобальный идентификатор транзакций — `std::atomic<uint64_t>` с `fetch_add`.
- Результат выполнения задачи передаётся воркером через `std::promise`: воркер вызывает `promise->set_value(final_tx)`, вызывающая сторона получает финальную транзакцию через `std::future::get()`.

### 2.2 Клиент

Реализован на C++ с использованием Qt Widgets.

**Многопоточность клиента:**
- UI-поток не блокируется сетевыми операциями.
- Все запросы к серверу выполняются в отдельном потоке (QThread); результат передаётся в UI через механизм signals/slots.
- Фоновый поток периодически запрашивает актуальный баланс.

## 4. Модель данных

### 4.1 Account

| Поле | Тип | Описание |
|------|-----|----------|
| `account_id` | uint64 | Уникальный идентификатор |
| `name` | string | Имя владельца |
| `type` | enum | `payer` или `merchant` |
| `balance` | int64 | Баланс в копейках |

### 4.2 Transaction

| Поле | Тип | Описание |
|------|-----|----------|
| `transaction_id` | uint64 | Из атомарного счётчика |
| `from_account_id` | uint64 | Аккаунт списания |
| `to_account_id` | uint64 | Аккаунт зачисления |
| `amount` | int64 | Сумма в копейках, > 0 |
| `status` | enum | `pending` / `completed` / `failed` |
| `fail_reason` | string | Причина отказа (если `failed`) |
| `created_at` | timestamp | Время создания |

Жизненный цикл статуса: транзакция создаётся со статусом `pending` в момент постановки в очередь, переходит в `completed` или `failed` после обработки воркером.

## 5. Функциональные требования

### FR-1. Регистрация аккаунта

Система создаёт аккаунт указанного типа и возвращает `account_id`.

Критерий приёмки: аккаунт доступен для операций сразу после создания.

---

### FR-2. Пополнение баланса

Система зачисляет указанную сумму на баланс аккаунта.

Ограничения: сумма должна быть строго больше нуля. Пополнение доступно только для аккаунтов типа `payer`; попытка пополнить аккаунт мерчанта возвращает ошибку `invalid_account_type`.

Критерий приёмки: баланс увеличивается ровно на указанную сумму.

---

### FR-3. Выполнение платежа

Система принимает запрос на перевод средств от плательщика к мерчанту и немедленно возвращает `tx_id` со статусом `pending`. Фактическое выполнение происходит асинхронно в пуле воркеров.

Требования:
- Запрос возвращает `tx_id` немедленно, не дожидаясь завершения обработки.
- Инициировать платёж может только аккаунт типа `payer`; попытка платежа от мерчанта возвращает ошибку `invalid_account_type`.
- Операция списания и зачисления атомарна: выполняется под `scoped_lock` на оба аккаунта.
- При недостаточном балансе транзакция переходит в статус `failed` с причиной `insufficient_funds`.
- При конкурентных запросах с одного аккаунта баланс не уходит в минус; каждая транзакция получает корректный финальный статус.

Критерий приёмки: в конкурентном тесте суммарное списание не превышает начальный баланс; все транзакции переходят из `pending` в корректный финальный статус.

---

### FR-6. Запрос статуса транзакции

Система возвращает текущий статус транзакции по её `tx_id`.

Критерий приёмки: статус корректно отражает состояние транзакции (`pending` / `completed` / `failed`).

---

### FR-4. Просмотр баланса

Система возвращает актуальный баланс аккаунта.

Критерий приёмки: значение соответствует состоянию после всех завершённых транзакций.

---

### FR-5. История транзакций

Система возвращает список транзакций, в которых участвовал аккаунт (как отправитель или получатель).

Критерий приёмки: список содержит все транзакции с корректными суммами и статусами.

---

## 6. Нефункциональные требования

### NFR-1. Потокобезопасность

- Отсутствие гонок данных при конкурентных платежах.
- Отсутствие deadlock при встречных переводах между одними и теми же аккаунтами.
- Консистентность балансов: сумма всех балансов в системе постоянна.

### NFR-2. Покрытие тестами

Минимальный набор тестов сервера:
- Успешный платёж при достаточном балансе.
- Отказ при нехватке средств.
- Два потока одновременно списывают с одного аккаунта — баланс не уходит в минус.
- Встречные переводы A→B и B→A из двух потоков — нет deadlock.
- Атомарный счётчик транзакций корректен при параллельных вызовах.

Минимальный набор тестов клиента:
- Валидация входных данных (сумма, идентификаторы).
- Корректная обработка ответов сервера (успех, ошибки).
- Модели данных Account и Transaction.

### NFR-3. Обработка ошибок

Сервер возвращает структурированные коды ошибок:

| Код | Описание |
|-----|----------|
| `invalid_request` | Некорректный формат запроса |
| `account_not_found` | Аккаунт не найден |
| `transaction_not_found` | Транзакция не найдена |
| `insufficient_funds` | Недостаточно средств |
| `invalid_amount` | Сумма не положительная |
| `invalid_account_type` | Операция недопустима для данного типа аккаунта |
| `internal_error` | Внутренняя ошибка сервера |

Клиент отображает все коды ошибок пользователю.

## 7. Сценарии использования

### UC-1: Регистрация и пополнение баланса

1. Пользователь вводит имя и тип аккаунта.
2. Клиент отправляет запрос `CreateAccount`.
3. Сервер создаёт аккаунт, возвращает `account_id`.
4. Пользователь вводит сумму пополнения.
5. Клиент отправляет запрос `Deposit`.
6. Сервер зачисляет средства, клиент отображает обновлённый баланс.

### UC-2: Выполнение платежа

1. Плательщик вводит `account_id` мерчанта и сумму.
2. Клиент отправляет запрос `PAY` в фоновом потоке.
3. Сервер создаёт транзакцию со статусом `pending`, сохраняет её, возвращает `tx_id` клиенту немедленно.
4. Задача помещается в `TaskQueue`; воркер извлекает её через `condition_variable`.
5. Воркер захватывает мьютексы обоих аккаунтов через `scoped_lock`.
6. Проверяется баланс; при достаточности выполняется списание и зачисление.
7. Воркер обновляет статус транзакции на `completed` или `failed`.
8. Клиент отправляет `GET_TX_STATUS|<tx_id>` и получает финальный статус.

### UC-3: Конкурентный платёж

1. Два клиента одновременно отправляют `PAY` с одного аккаунта.
2. Оба запроса немедленно получают `tx_id` со статусом `pending`.
3. Обе задачи попадают в `TaskQueue`; воркеры разбирают их параллельно.
4. Первый воркер захватывает `scoped_lock` на оба аккаунта, выполняет списание, обновляет статус на `completed`.
5. Второй воркер ожидает освобождения мьютекса, затем проверяет баланс — средств недостаточно.
6. Второй платёж получает статус `failed`, причина `insufficient_funds`.

---

## 8. Definition of Done

- Сервер и клиент запускаются независимо и взаимодействуют по сети.
- Сервер многопоточен: пул потоков, очередь задач, потокобезопасное хранилище.
- Клиент многопоточен: UI не блокируется при сетевых операциях.
- Реализованы FR-1 — FR-6.
- Все unit-тесты проходят (`ctest --output-on-failure`).
- Конкурентный тест подтверждает корректность при параллельных платежах.
- Встречные переводы не приводят к deadlock.
- Проект собирается через CMake без ошибок.
